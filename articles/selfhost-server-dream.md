## ガチ自家サーバへの憧れ

自宅にラックを組んで、強い回線引いて、24時間ファンの音が響く部屋を作る。それがしばらくは私の憧れでした。なんか特殊で難しいことしてる、機械に囲まれている、そんなシンプルな憧れでした。

恐らく業務用のマジなラックマウント PC は良いお値段がする。それらを買うのは難しいだろう。ならばせめて、ラックに取り付けられる形の自作 PC を作れないか？ラックマウントの寸法なんてよく出回っているから、ちょっと CAD とかを使えればいけるだろう。

そんなことを夢見ていたわけですが･･･5年くらいしてサーバ管理の現実を知るわけです。

## 本気のサーバ管理とは

### HTTPS (SSL 対応) はほぼ義務

Web サーバを建て、それをオンラインで公開するにあたっては SSL 対応、つまり HTTP でなく HTTP**S** にするのが現代ではほぼ義務です。HTTPS でないサイトへのアクセス時、多くのブラウザではデフォルトで警告が出るようになってしまいました。

HTTPS 化のやり方は調べればすぐ見つかります。「じゃあ対応すればいいじゃん」ということになるわけですが･･･

HTTPS 化には証明書が必要になります (技術的な詳細解説はここではカット)。正攻法としては外部機関、つまり自分以外の誰かに証明書を出してもらう必要があります。そしてゲットした証明書をあれやこれや設定してと、やらねばならんことがあるわけです。プロジェクト立ち上げの熱い想いがあれば一応乗り越えられるかもしれません。

:::message
自己署名証明書、いわゆるオレオレ証明書でも一応 HTTPS 化はできます。ただし、当然これではほぼ全てのブラウザで警告が出ます。本気のサーバ建てではまかり通りません。
:::

なんとか乗り越えて HTTPS 化できたとして、さらに次の問題。通常、発行される証明書には有効期限があります。数年単位の一定期間ごとに証明書を作りなおさなければいけません。セキュアを保つためには定期的に更新しなければならないのです。会社とかなら専門部署で管理してくれるでしょう。しかし自家サーバでは自分で全てやるしかありません。更新を忘れて有効期限が切れれば、たちまち警告が勃発することでしょう。

### OS などのアップデート

OS やライブラリ、フレームワークを使う以上、脆弱性とは縁を切れません。かといってそれら無しにアプリやサーバを作ることは困難です。脆弱性のリスクをなるべく回避するには、絶えずシステムをアップデートする必要があります。

ただ、何も考えず全て最新にアップデートすれば良いかというと、話はそう単純ではありません。開発するエンジニアも人間ですから、ミスをすることはあります。最新バージョンにとんでもないミスがあることだって起こり得ます。

使っているシステムとバージョンを把握し、それらのアップデート情報やトラブル情報にアンテナを張り続けなければいけないのです。

### ネットワーク機器の調整

サーバを構築するという仕事は、当然ながらスマホを Wi-Fi につなげるのとは訳が違います。ネットワーク機器およびサーバ PC のネットワーク設定を色々いじらなければなりません。

ポート転送、ファイアウォール、IP アドレス･･･。そして機械は実に素直で融通がききません。自分の手がミスれば機械の動作もミスります。最悪そこがシステムの脆弱性にもなり得ます。「これだと動かないよ？ヤバくなるよ？」と教えてくれれば嬉しいですが、そんな優しい機器もそう無いでしょう。

ネットワーク機器を一元管理するシステムを使えるなら多少話は変わるかもしれません。ただ、それを導入して扱うにもそれなりの知識が要求されることでしょう。

### 重くのしかかる「セキュリティ」

ここまでの話でも触れてきた通り、セキュリティのために意識しないといけないことが実にたくさんあるわけです。仮に使っている OS やネットワーク機器たちが脆弱性のない完璧な状態だったとしても、設定でヘマして自分の手で脆弱性を作ってはおしまいです。

この仕事では、パスワードや秘密鍵ファイルなど、漏洩してはいけないデータというのが必然的に生じてきます。間違ってでも外部からそれらに到達可能になってはいけません。しかしシステムが複雑になってくると、それも自分 1人だけのチェックだと、ついうっかり･･･そして融通のきかない機械はそれを通してしまいます。

OS や機器など、基幹となるシステムの設定だけではありません。自分の手で構築するアプリケーションもそうです。例えば BBS や SNS 的なサービスを作ったとして、投稿テキストのエスケープを忘れて [XSS](https://www.ipa.go.jp/security/vuln/websecurity/cross-site-scripting.html) を可能にしてしまった！みたいな。

### 24時間稼働の '保証'

世の多くの Web サービスは 24時間営業です。朝早くでも真っ昼間でも深夜でも、アクセスすればいつでもサービスを受けられます。いつでもサービスを提供するには、当然サーバが 24時間 365日稼働していなければなりません。そしてそのサーバおよびその周辺機器というのは･･･

- 電源が切れれば止まります。しかし停電が一切起きないと保証された地域は存在しません
- アップデートでシステムが狂えば止まります。しかしアップデートしなければ、脆弱性がそのままだったり古い機能しか使えなかったりします
- ハードウェアが寿命で壊れれば止まります。しかし寿命がなく壊れないハードウェアは存在しません
- ハードウェアが自然災害で壊れれば止まります。しかし災害ゼロの絶対安全地帯は地球上に存在しません

それぞれに対して可能な対策はあります。ただし対策を完璧に近づけるのは非常に膨大な労力が要るか、あるいは個人レベルでは到底困難なケースもままあります。

## ホスティングサービスっていいよね

こうして考えてみると、レンタルサーバやホスティングサービスは月額制だとしてもとても嬉しいことがわかります。

まずハードウェアは保証してくれます。大手が何十億、何百億円とかかけて整備したであろうサーバ施設・データセンターを使えます。バックアップもあるでしょう。ハードウェア故障に怯える必要はそうありません。

ネットワーク機器設定も、開けるポートとか、脆弱性でなく本来設定したいことに集中できます。また、サービスによっては HTTP サーバの SSL 化が標準装備というのも今や珍しくありません。アプリケーション作りに集中できます。

もちろん、そこに任せてれば完璧かというと、そうではありません。AWS とかの有名どころだって障害を起こして止まることはあります。それでも自分 1人、または頑張っても友人 3人程度とかで管理できる規模と比べればはるかに盤石でしょう。

ホスティングサービスに限らず、GitHub とか YouTube とか、それらサービスだってハードウェアや 24時間稼働を十分保証してくれているわけです。ホスティングサービスに限らず、誰かがサーバ管理をしてくれているあらゆるものが、ありがたく感じてきませんか？

## ただ、ハードウェアを触ることも価値である

では、本気で自家サーバをやるのは価値が無いのか？いいえ、そうは思いません。

自家サーバは、機械を触れるという楽しみがあります。レンタルサーバやホスティングサービスでは当然全て画面上操作なので、機械を触ることなんてありません。それが便利でもあり、一方でハードウェアを触る楽しみがないとも言えるわけです。

あらゆるものに触れる、つまり体験するというのは、最高の勉強にもなります。やったもん勝ちです。

全てのシチュエーションにおいて完璧なシステムである必要は全くありません。友人 3人とのマイクラサーバを自宅に建てるのに、データバックアップと非常用電源が要るでしょうか。とんでもないこだわりでさえなければ、普通の PC 1台、あるいは用途次第ではラズパイ 1枚程度でも十分です。

小規模だったとしても、ポート解放、ポート転送、ファイアウォール設定などをちょっとやってみるだけで、学べることはあるはずです。どっちも否定しませんし、否定できません。

サービス作りに集中したくて、ハードウェア触りが主でないならレンタルサーバやホスティングサービス。ハードウェア触りを楽しみたいなら、お財布と相談しながら存分に機器を買ってシステム構築。どっちも正当であり、下位でも上位でもありません。

私はサービス作りに集中したかったことに気付いたので、本気のサーバ作りには至らなかったわけです。
